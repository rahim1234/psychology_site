# مرکز مشاوره روانشناسی (Django)

سایت مدیریت جلسات روانشناسی: ثبت‌نام کاربر با تایید ایمیل، تکمیل پروفایل و ثبت مشکل،
انجام اختیاری آزمون‌های PHQ-9 (افسردگی) و GAD-7 (اضطراب)، و پنل مدیریت (درمانگر) برای
مشاهده‌ی نتایج و ثبت یادداشت/خلاصه‌ی هر جلسه که فقط برای درمانگر قابل مشاهده است.

## راه‌اندازی (Development)

```bash
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

cp .env.example .env
# .env را باز کنید و SECRET_KEY را با یک مقدار تصادفی جایگزین کنید:
python3 -c "from django.core.management.utils import get_random_secret_key; print(get_random_secret_key())"

python manage.py migrate
python manage.py createsuperuser   # حساب درمانگر/ادمین
python manage.py runserver
```

در حالت DEBUG=True، ایمیل‌ها (از جمله کد تایید ثبت‌نام) به‌جای ارسال واقعی، در ترمینال
چاپ می‌شوند (`console` email backend)، پس نیازی به تنظیم SMTP برای تست محلی نیست.

## راه‌اندازی (Production)

مقادیر واقعی را در `.env` قرار دهید (نمونه در `.env.example`):

- `SECRET_KEY` تصادفی و طولانی
- `DEBUG=False`
- `ALLOWED_HOSTS` = دامنه‌ی واقعی سایت
- `EMAIL_BACKEND=django.core.mail.backends.smtp.EmailBackend` به همراه
  `EMAIL_HOST` / `EMAIL_PORT` / `EMAIL_HOST_USER` / `EMAIL_HOST_PASSWORD` / `EMAIL_USE_TLS`
  و `DEFAULT_FROM_EMAIL` (اطلاعات SMTP واقعی خودتان، مثلاً Zoho، Mailgun، SendGrid و ...)

سپس `deploy.sh` را روی سرور اوبونتو اجرا کنید. این اسکریپت دیگر پسورد ادمین را در کد
هاردکد نمی‌کند؛ در پایان از شما به‌صورت تعاملی برای ساخت حساب ادمین (`createsuperuser`)
سوال می‌پرسد.

## تغییرات و اصلاحات نسبت به نسخه‌ی قبلی

### امنیت

1. **تایید ایمیل هنگام ثبت‌نام (درخواست اصلی شما):** حساب کاربری بعد از ثبت‌نام
   غیرفعال (`is_active=False`) ساخته می‌شود، یک کد ۶ رقمی تصادفی (با ماژول امن `secrets`،
   نه `random`) تولید و **هش‌شده** (مثل پسورد) در دیتابیس ذخیره می‌شود، و کد از طریق ایمیل
   ارسال می‌گردد. کد ۱۰ دقیقه اعتبار دارد، حداکثر ۵ بار قابل تلاش است و امکان "ارسال مجدد
   کد" با فاصله‌ی ۶۰ ثانیه‌ای (جلوگیری از اسپم) وجود دارد. همه‌ی این عددها در `.env` قابل
   تنظیم هستند.
2. **یکتا شدن ایمیل کاربران:** قبلاً فیلد ایمیل `unique` نبود و امکان ثبت‌نام چند حساب با
   یک ایمیل (و حتی کرش بک‌اند ورود) وجود داشت؛ اصلاح شد.
3. **یادداشت‌های جلسه (Session Notes) برای درمانگر:** مدل جدید `SessionNote` اضافه شد که
   فقط از طریق پنل مدیریت (`/admin/`) قابل مشاهده و ویرایش است — یعنی فقط کاربرانی که
   `is_staff` هستند (درمانگر) به آن دسترسی دارند. هیچ‌کدام از صفحات کاربری (داشبورد،
   پروفایل و ...) این یادداشت‌ها را نمایش نمی‌دهند.
4. **رمزهای پیش‌فرض هاردکد‌شده در `deploy.sh` حذف شد:** نسخه‌ی قبلی یک نام‌کاربری/پسورد
   ادمین ثابت (`rahim@example.com` / `R@him2024!StrongP@ss`) داخل اسکریپت commit شده بود؛
   این یک ریسک امنیتی جدی است چون هرکسی که کد را ببیند آن پسورد را هم می‌بیند. الان
   `createsuperuser` به‌صورت تعاملی اجرا می‌شود.
5. **باگ اسکریپت `deploy.sh`:** تولید `SECRET_KEY` به‌خاطر quote اشتباه در heredoc اصلاً
   اجرا نمی‌شد و متن دستور به‌صورت خام در `.env` نوشته می‌شد؛ اصلاح شد.
6. **`AUTH_PASSWORD_VALIDATORS`:** که در نسخه‌ی قبلی اصلاً تعریف نشده بود (یعنی کاربر
   می‌توانست پسورد ۱ کاراکتری بگذارد) اضافه شد (حداقل طول، جلوگیری از پسورد رایج/عددی و...).
7. **تنظیمات امنیتی Production:** `SECURE_SSL_REDIRECT`, `SESSION_COOKIE_SECURE`,
   `CSRF_COOKIE_SECURE`, `SECURE_HSTS_*`, `X_FRAME_OPTIONS`, `CSRF_TRUSTED_ORIGINS` اضافه شد
   (پیش‌فرض‌شان بر اساس `DEBUG` تنظیم می‌شود، و از طریق `.env` هم قابل override هستند).
8. **`SECRET_KEY`:** دیگر یک مقدار پیش‌فرض ثابت و ناامن در کد نیست؛ اگر در حالت
   `DEBUG=False` مقداردهی نشود، سرور اصلاً بالا نمی‌آید (به‌جای اجرای خاموش و ناامن).
   `DEBUG` هم پیش‌فرضش `False` شد (قبلاً پیش‌فرض `True` بود که برای Production خطرناک است).
9. **خروج از حساب (Logout) با POST به‌جای GET:** لینک خروج قبلی یک درخواست GET بود که
   طبق best-practiceهای Django/OWASP نباید باعث تغییر state (خروج از سشن) شود؛ الان یک
   فرم POST با CSRF token است.
10. **جلوگیری از دسترسی به مقالات منتشرنشده‌ی وبلاگ:** قبلاً هرکسی با حدس زدن لینک مستقیم
    یک مقاله‌ی `published=False` می‌توانست آن را ببیند؛ الان فقط نویسنده/ادمین (`is_staff`)
    می‌تواند پیش‌نمایش بگیرد.
11. **حفاظت در برابر Timing Attack ساده در ورود:** حتی وقتی ایمیل وجود ندارد، بک‌اند
    احراز هویت همان مقدار محاسبات هش پسورد را انجام می‌دهد تا زمان پاسخ اطلاعاتی درباره‌ی
    وجود/عدم‌وجود ایمیل درز نکند.
12. **پیام خطای ورود:** به‌جای پیام گیج‌کننده‌ی عمومی، اگر حساب هنوز تایید نشده باشد کاربر
    راهنمایی می‌شود که ایمیلش را برای کد تایید بررسی کند (بدون افشای اینکه دقیقاً کدام
    ایمیل‌ها در سیستم ثبت شده‌اند).

### باگ‌ها

- در `ProfileUpdateView.post` اگر کاربر بدون داشتن پروفایل مستقیماً POST می‌زد، خطای
  ۵۰۰ می‌گرفت؛ اصلاح شد.
- در `PHQ9TestView`/`GAD7TestView` بعد از ساخت نتیجه، یک کوئری اضافه و غیرضروری برای پیدا
  کردن «آخرین نتیجه» زده می‌شد؛ چون خودِ `objects.create()` شیء ساخته‌شده را برمی‌گرداند،
  کوئری تکراری حذف شد.
- در `blog` مدل `Post` فیلد `author` نداشت ولی کد View سعی می‌کرد `form.instance.author`
  را ست کند (این مقدار هرگز ذخیره نمی‌شد)؛ فیلد `author` واقعی به مدل اضافه شد.
- `import` اشتباه و درون‌متنیِ `from django.shortcuts import render` وسط
  `profiles/views.py` که فقط به‌خاطر شانس کار می‌کرد، جابه‌جا و مرتب شد.

### پاکسازی کد (Clean-up)

- فایل `compat_patch.py` (یک monkey-patch عجیب برای «سازگاری با پایتون ۳.۱۴») حذف شد.
  Django 5.1 رسماً از پایتون ۳.۱۴ پشتیبانی نمی‌کند و این پچ صرفاً یک دورزدن ناپایدار بود؛
  توصیه می‌شود از Python 3.11 یا 3.12 استفاده کنید (در `requirements.txt` یادداشت شد).
- import های پراکنده و اضافی در `assessments/forms.py` به بالای فایل منتقل شدند.
- کوئری‌های تکراری و کد مرده حذف شدند (بالا توضیح داده شد).

## نکته‌ی مهاجرت داده (Migration)

فیلد ایمیل حالا `unique` است. اگر روی یک دیتابیس *قدیمی* که از قبل رکوردهای تکراری ایمیل
دارد migrate بزنید، migration به خطا می‌خورد. برای دیتابیس تازه (پروژه‌ی جدید) مشکلی
نیست و مستقیم `python manage.py migrate` کافی است.

## معماری «یادداشت جلسه»

`SessionNote` (در `profiles/models.py`) به هر `Profile` متصل است و شامل تاریخ جلسه، متن
یادداشت/خلاصه، و اینکه چه کسی (کدام ادمین) آن را ثبت کرده. مدیریت آن از طریق پنل ادمین
Django روی صفحه‌ی همان پروفایل کاربر انجام می‌شود (inline)، دقیقاً کنار نتایج آزمون‌های
PHQ-9/GAD-7 که از قبل آنجا نمایش داده می‌شدند. این محتوا هیچ‌وقت در هیچ View یا Templateی
که کاربر عادی می‌بیند خوانده نمی‌شود.

فقط به حساب‌هایی که `is_staff=True` هستند دسترسی پنل ادمین بدهید (یعنی فقط درمانگر/درمانگران).

## اجرای سایت بدون اینترنت (Offline)

نسخه‌ی اولیه‌ی پروژه، بوت‌استرپ (ظاهر/منو) و آیکون‌ها را از یک CDN اینترنتی
(`cdn.jsdelivr.net`) لود می‌کرد. روی کامپیوتری که اینترنت ندارد، این باعث کند شدن شدید
بارگذاری صفحه و به‌هم‌ریختن ظاهر/منو می‌شود (چون فایل CSS اصلاً لود نمی‌شود) — پنل ادمین
چون فایل‌های استاتیک خودش را از داخل جنگو می‌گیرد، مشکلی نداشت.

اصلاح شد تا این فایل‌ها به‌صورت لوکال (از پوشه‌ی `static/vendor/`) لود شوند. چون این
فایل‌ها باینری/حجیم هستند، خودِ فایل‌ها داخل این تحویل قرار نگرفته‌اند و باید یک‌بار،
روی یک کامپیوتر که اینترنت دارد، دانلود و در مسیرهای زیر کپی شوند:

```
static/vendor/bootstrap/bootstrap.min.css
static/vendor/bootstrap/bootstrap.bundle.min.js
static/vendor/bootstrap-icons/bootstrap-icons.css
static/vendor/bootstrap-icons/fonts/bootstrap-icons.woff2
```

لینک‌های دانلود:

- https://cdn.jsdelivr.net/npm/bootstrap@5.3.0/dist/css/bootstrap.min.css
- https://cdn.jsdelivr.net/npm/bootstrap@5.3.0/dist/js/bootstrap.bundle.min.js
- https://cdn.jsdelivr.net/npm/bootstrap-icons@1.10.0/font/bootstrap-icons.css
- https://cdn.jsdelivr.net/npm/bootstrap-icons@1.10.0/font/fonts/bootstrap-icons.woff2

بعد از کپی کردن فایل‌ها، برای Production حتماً یک‌بار این دستور را اجرا کنید تا فایل‌ها
در `STATIC_ROOT` جمع‌آوری شوند (روی `runserver` محلی لازم نیست):

```bash
python manage.py collectstatic --noinput
```

## پیوست عکس/صدا/ویدیو و تاریخ شمسی در یادداشت‌های جلسه

هر یادداشت جلسه (`SessionNote`) الان سه فیلد اختیاری اضافه دارد: `image` (عکس، حداکثر
۱۰ مگابایت)، `audio` (صدا — فرمت‌های mp3/wav/ogg/m4a/aac، حداکثر ۲۰ مگابایت)، `video`
(ویدیو — فرمت‌های mp4/mov/webm/avi، حداکثر ۱۰۰ مگابایت). این‌ها از همان فرم افزودن
یادداشت در صفحه‌ی پروفایل هر کاربر در پنل ادمین قابل آپلود هستند، و بعد از ذخیره،
پیش‌نمایش/پخش‌کننده‌شان هم همان‌جا نمایش داده می‌شود.

تاریخ جلسه (`session_date`) در پنل ادمین با یک ورودی سه‌قسمتی (سال/ماه/روز شمسی) وارد
و نمایش داده می‌شود. تبدیل تاریخ شمسی↔میلادی با یک الگوریتم استاندارد و کاملاً به زبان
پایتون خالص در `profiles/jalali.py` انجام می‌شود — **هیچ پکیج خارجی (pip) یا فایل
اینترنتی لازم نیست**؛ این تصمیم عمدی بود چون قبلاً هم به مشکل نصب پکیج و هم به مشکل CDN
بدون اینترنت برخورده بودید. در پایگاه‌داده، تاریخ همچنان به شکل استاندارد میلادی ذخیره
می‌شود (بدون نیاز به تغییر نوع فیلد یا مهاجرت پیچیده).

### نصب
هیچ پکیج جدیدی لازم نیست. فقط کافیست migration جدید را اجرا کنید:
```bash
python manage.py migrate
```

### فیلد «تاریخ فایل» (جدید) در پیوست‌های پروفایل
مدل `ProfileAttachment` الان یک فیلد `upload_date` (تاریخ شمسی) دارد که منشی هنگام
آپلود فایل آن را مشخص می‌کند — یعنی تاریخی که فایل به آن تعلق دارد (مثلاً تاریخ جلسه یا
تاریخ سند)، نه لزوماً همین الان. در پایگاه‌داده این فیلد به‌صورت میلادی ذخیره می‌شود ولی
در پنل ادمین با همان ویجت سه‌قسمتی سال/ماه/روز شمسی وارد و نمایش داده می‌شود. برای
دیتابیس‌های قدیمی، migration به‌صورت خودکار تاریخ امروز را برای ردیف‌های موجود قرار
می‌دهد تا بدون خطا اجرا شود.

### نکته‌ی حجم آپلود روی سرور Production
چون فایل ویدیو می‌تواند تا ۱۰۰ مگابایت باشد، مقدار `client_max_body_size` در
`nginx.conf` به ۱۱۰ مگابایت افزایش یافت. اگر از قبل روی سرور دیپلوی کرده بودید،
این تغییر را باید **دستی** هم در `/etc/nginx/sites-available/psychology_site` اعمال
کنید و `sudo systemctl restart nginx` بزنید (چون `deploy.sh` این فایل را فقط بار اول
کپی می‌کند، نه هر بار).

### امنیت این فایل‌ها (بعداً به‌روزرسانی شد — الان محافظت‌شده هستند)
فایل‌های پیوست (عکس/صدا/ویدیوی یادداشت جلسه، و فایل‌های پیوست پروفایل) دیگر زیر
`MEDIA_URL` عمومی ذخیره نمی‌شوند. یک محل ذخیره‌سازی کاملاً جدا (`PRIVATE_MEDIA_ROOT`)
دارند که نه nginx و نه حالت DEBUG جنگو مستقیم آن را serve نمی‌کنند. تنها راه دسترسی به
این فایل‌ها یک View محافظت‌شده (`profiles.views.protected_media`) است که:
- کاربر باید لاگین کرده و `is_staff` باشد (درمانگر یا منشی)، وگرنه به صفحه‌ی ورود ادمین
  هدایت می‌شود؛
- علاوه بر آن، مجوز Django مخصوص همان مدل را هم چک می‌کند — یعنی منشی چون مجوز
  `view_sessionnote` ندارد، حتی اگر لینک فایل یک یادداشت پزشک را حدس بزند/دوباره از آن
  استفاده کند، با خطای دسترسی (403) مواجه می‌شود.

⚠️ فایل‌هایی که **قبل از این به‌روزرسانی** آپلود کرده بودید، در مسیر عمومی قدیمی
(`media/session_notes/...`) باقی مانده‌اند و همچنان با لینک مستقیم قابل‌دسترسی‌اند. اگر
فایل حساسی از قبل آپلود کرده بودید، بهتر است آن یادداشت را دوباره باز کنید، فایل قدیمی
را حذف و دوباره آپلود کنید تا در محل جدید و امن ذخیره شود.

## نقش «منشی» — دسترسی محدود برای آپلود فایل

منشی می‌تواند:
- در لیست پروفایل‌ها فقط **نام** کاربران را ببیند و روی نام کلیک کند تا وارد پروفایل شود.
- در صفحه‌ی هر پروفایل، فقط یک فرم آپلود فایل ببیند که شامل **فایل**، **تاریخ (شمسی)** و توضیح است.
- فایل را به همراه تاریخی که خودش مشخص می‌کند (مثلاً تاریخ جلسه یا تاریخ سند) آپلود کند.

منشی **نمی‌تواند**:
- یادداشت‌های جلسه‌ی درمانگر (`SessionNote`) را ببیند — این بخش اصلاً برایش نمایش داده
  نمی‌شود و دسترسی URL مستقیم هم به آن ندارد.
- نتایج آزمون‌های PHQ-9/GAD-7 را ببیند — نه در لیست پروفایل‌ها (ستون‌های شمارش نتایج
  مخفی است) و نه در صفحه‌ی جزئیات پروفایل.
- اطلاعات قبلی کاربر را ببیند — سن، جنسیت، توضیح مشکل و تاریخ ایجاد پروفایل برایش
  مخفی است؛ فقط نام کاربر را می‌بیند.
- فایل‌های قبلی‌ای که خود دکتر یا منشی‌های دیگر آپلود کرده‌اند را ببیند — فقط فرم آپلود
  فایل جدید را می‌بیند (نه لیست فایل‌های موجود را).
- هیچ پروفایل یا کاربری را حذف یا اضافه کند.
- فایلی که خودش یا دیگری آپلود کرده را ویرایش/جایگزین/حذف کند (فقط افزودن).
- به بخش‌های وبلاگ یا حساب‌های کاربری (`accounts.User`) دسترسی داشته باشد.

### راه‌اندازی یک‌بار (توسط شما، به‌عنوان ادمین اصلی)

#### ۱. ساخت گروه «منشی» با مجوزهای صحیح

به‌جای ساخت دستی گروه در پنل ادمین، کافیست این دستور را یک‌بار اجرا کنید:

```bash
python manage.py setup_secretary_group
```

این دستور:
- گروهی به نام دقیق **`منشی`** می‌سازد (اگر نباشد).
- دقیقاً ۴ مجوز زیر را به آن می‌دهد (نه بیشتر، نه کمتر):
  - `profiles | profile | Can view profile`
  - `profiles | profile | Can change profile` (فقط برای ذخیره‌ی فرم آپلود فایل لازم است)
  - `profiles | profile attachment | Can view profile attachment`
  - `profiles | profile attachment | Can add profile attachment`

اجرای دوباره‌ی این دستور بی‌خطر است؛ اگر گروه از قبل وجود داشته باشد، مجوزها را به
حالت استاندارد بازنشانی می‌کند (یعنی اگر کسی دستی مجوز اضافه‌ای داده بود، حذف می‌شود).

#### ۲. ساخت حساب کاربری برای منشی

یک حساب کاربری برای منشی بسازید (یا حساب موجودش را ویرایش کنید) در
`/admin/accounts/user/add/` (یا `/admin/accounts/user/<id>/change/`):
- تیک **Staff status** را بزنید (بدون این، اصلاً نمی‌تواند وارد پنل ادمین شود).
- تیک **Superuser status** را **نزنید** (این خیلی مهم است — superuser از همه‌ی
  محدودیت‌ها عبور می‌کند).
- در بخش «Groups»، گروه **منشی** را انتخاب و به سمت راست منتقل کنید.
- رمز عبور را تنظیم کنید و ذخیره کنید.

از این به بعد، وقتی منشی با این حساب وارد `/admin/` شود:
- فقط بخش «پروفایل‌ها» را می‌بیند (نه وبلاگ، نه کاربران، نه آزمون‌ها).
- داخل «پروفایل‌ها» فقط **نام** کاربران را می‌بیند — نه سن، نه جنسیت، نه نتایج آزمون.
- روی نام کاربر کلیک می‌کند و فقط فرم آپلود فایل (به همراه تاریخ شمسی) را می‌بیند.
- بعد از ذخیره، فایل با تاریخ مشخص‌شده ذخیره می‌شود و `uploaded_by` خودکار روی حساب
  منشی تنظیم می‌شود.
